<figure><img alt="" src="https://cdn-images-1.medium.com/max/1000/1*9CebBd8xHWQgGnwd1fbVEA.jpeg" /></figure><p>For several years, Xero has maintained two versions of the Xero Practice Manager (XPM) API: v3.0 (Integer-based) and v3.1 (UUID-based). To ensure we can continue to scale our infrastructure and provide better performance, we are now beginning the process of consolidating onto a single version.</p><p><strong>Migrate to v3.1 by April 30, 2027, to unlock greater integration efficiency and features</strong></p><p>While v3.1 has been available for some time, this 12-month notice marks the start of the formal deprecation period for v3.0. We are encouraging all app partners and developers to migrate to v3.1 to take advantage of improved data handling and to ensure service continuity.</p><h3>Why migrate to v3.1 now?</h3><p>Beyond the April 2027 deadline, v3.1 offers significant functional advantages that help you build more efficient integrations:</p><ul><li><strong>JSON Support:</strong> On Practice Manager 3.1, endpoints support both XML and JSON. You choose the format using standard Accept and Content-Type headers, and JSON responses use camelCase field names. See <a href="https://developer.xero.com/documentation/api/practice-manager-3-1/content-negotiation">documentation </a>for more detail.</li><li><strong>Payload Optimisation (Field Selection):</strong> On v3.1, a fields query parameter lets you request just the data you need, and the <a href="https://developer.xero.com/documentation/api/practice-manager-3-1/field-selection">docs highlight</a> how to use it across core areas like Clients, Jobs, Staff, and Custom Fields.</li><li><strong>Managing “Data Out” Costs:</strong> As Xero moves toward usage-based models for data transfer, the ability to limit your payload size via Field Selection and JSON will be a critical tool for managing your integration’s operational costs.</li><li><strong>Expanded Data Sets:</strong> v3.1 includes the new <strong>Job States</strong> endpoint and several new Client fields to ensure parity with the latest XPM UI features:</li><li><strong>New Demographic Fields:</strong> Support for DateOfDeath and Place of Birth (City and CountryCode) for individual and sole trader clients.</li><li><strong>UK Region Updates:</strong> Dedicated fields for MiddleName and Title for UK-based individual and sole trader clients.</li><li><strong>Universal Identifiers:</strong> A dedicated Client Reference Code field (now available across all regions) and read-only Industry / IndustryTitle labels.</li><li><strong>OpenAPI Spec &amp; API Explorer (Next Week):</strong> We are currently developing an OpenAPI specification for the XPM API v3.1. This will bring v3.1 into our <strong>API Explorer for JSON-based requests</strong>, providing an interactive sandbox to test endpoints and view schemas. Please note that the API Explorer will exclusively support JSON, further streamlining the development process for partners moving away from XML.</li></ul><h3>Your Migration Path</h3><p>Because v3.1 has existed alongside v3.0, many of you may already be seeing UUIDs in your current payloads. The migration is designed to be low-friction:</p><ol><li><strong>Map your IDs:</strong> v3.0 responses already include UUID fields. If you haven’t already, begin persisting these UUIDs in your database now.</li><li><strong>Update your Base URL:</strong> Switch your requests from /practicemanager/3.0/ to /practicemanager/3.1/.</li><li><strong>Update your Paths:</strong> Replace Integer IDs in your URL paths with the corresponding UUIDs.</li><li><strong>No Scope Changes:</strong> Your existing OAuth 2.0 scopes remain valid. No re-authentication or permission changes are required for this move.</li></ol><h3>Key Dates to Track</h3><ul><li><strong>Now:</strong> v3.1 is fully operational and the recommended version for all new development.</li><li><strong>April 30, 2027:</strong> v3.0 will be retired and no longer accept traffic.</li></ul><h3>Next Steps</h3><ul><li><strong>Identify V3.0 Traffic:</strong> To see if you are still using v3.0, visit the <strong>Xero Developer Console</strong>, select your app, and navigate to the <strong>Logs</strong> tab. Use the <strong>Filter</strong> tool to search for <strong>“3.0”</strong> in the URL field. This will instantly identify any active requests still hitting the V3.0 endpoints.</li></ul><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*y42NNUAg-JrpnxWP_wCDbA.png" /></figure><ul><li><strong>Start storing UUIDs:</strong> v3.0 responses already include UUID fields. Begin persisting these in your database now to make your eventual switch a simple ID swap.</li><li><strong>Test the switch:</strong> Plan and implement the base URL change to /practicemanager/3.1/ in your demo environment to verify your integration’s data handling.</li><li><strong>Roll out to production:</strong> Complete your migration well ahead of the <strong>April 30, 2027</strong> deadline to ensure zero service disruption.</li></ul><img alt="" height="1" src="https://medium.com/_/stat?event=post.clientViewed&amp;referrerSource=full_rss&amp;postId=94bd054ee7f6" width="1" /><hr /><p><a href="https://devblog.xero.com/consolidating-the-xero-practice-manager-api-moving-from-v3-0-to-v3-1-94bd054ee7f6">Consolidating the Xero Practice Manager API: Moving from v3.0 to v3.1</a> was originally published in <a href="https://devblog.xero.com">Xero Developer</a> on Medium, where people are continuing the conversation by highlighting and responding to this story.</p>
